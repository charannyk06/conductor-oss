# Conductor Bridge — PRD v2

**Revised after Mar 20 design review with Charann**

> Archive note
> This was a proposed hosted bridge model. The shipped product instead remains local-first by default, with optional bridge/relay access-control flows and a multi-agent dashboard experience. Use this doc as historical context only.

## What It Is

This was a proposed hosted bridge model. The shipped product instead keeps the dashboard local-first by default, forwards backend URLs into the web app through the launcher, and layers optional bridge/relay access-control flows on top. Users still run coding agents locally on their own machine.

The bridge is a background service on the user's laptop that pairs **once** with their account and stays connected forever.

## The Flow

```
User's Laptop (local)                 Hosted Dashboard (cloud)
─────────────────────                  ──────────────────────────
conductor-bridge daemon
│  starts on boot / login
│  auto-connects to relay
│
│  [first time only]
│  → opens browser to dashboard/bridge/pair
│  → user clicks "Connect this laptop"
│  → bridge receives device token (via localhost)
│  → stores refresh token permanently
│  → connects to relay
│
│                                    [dashboard shows]
│                                      "MacBook Pro — Online"
│
│  [any session/agent activity]
│  ←─────────────────────────────  user works via dashboard UI
│
│  agents run locally on laptop
│  terminal streams to dashboard
│  all data stays on laptop
```

## Core Properties

- **Pair once, done forever** — after first pairing, bridge auto-connects on every boot
- **All work on user's machine** — agents, terminal, files all run locally
- **Universal web access** — log in from any device to use your laptop's agents
- **No repeated auth** — refresh token stored locally, auto-renews
- **User's relay, user's rules** — self-hosted relay passes only encrypted bytes

## Architecture

```
[Any Device] ←——HTTPS/WSS——— [Hosted Dashboard + Relay] ←——WSS——— [Bridge on Laptop]
  browser                              (deployed at              laptop runs
  logs in here                          relay.conductor.dev)     agents locally
```

### Components

1. **Bridge binary** (`conductor-bridge`)
   - Background daemon (no UI window)
   - Runs on laptop startup (launchd/systemd)
   - Connects outbound to relay (firewall/NAT friendly)
   - Handles device pairing flow
   - Maintains WebSocket tunnel to relay
   - Proxies terminal, API, and file access to dashboard

2. **Relay server**
   - Hosted by Charann (or self-hosted by user)
   - WebSocket relay: wires browser sessions to bridge sessions
   - Device authorization server (device code OAuth flow)
   - Token storage (refresh tokens mapped to device IDs)
   - Zero session data — stateless pipe

3. **Dashboard (web)**
   - GitHub OAuth login
   - "Add a laptop" wizard (first-time pairing)
   - Live status of all paired devices
   - Terminal, agent management, file browser — all via bridge tunnel

## Pairing Flow (First Time)

```
User opens dashboard
  ↓
GitHub OAuth login
  ↓
Dashboard: "Add a laptop" button
  ↓ (user clicks it)
Dashboard generates pairing code: MACBOOK-7X2
  ↓
User runs on laptop:
  $ conductor-bridge pair --code MACBOOK-7X2
  ↓
Bridge → POST /api/devices/pair { code: "MACBOOK-7X2" }
Relay validates code, creates device record
Relay returns refresh token to bridge
Bridge stores refresh token permanently at ~/.conductor/bridge-refresh-token
Bridge connects to relay with device credentials
  ↓
Dashboard shows: "MacBook Pro — Connected"
```

**Key invariant:** Pairing happens AFTER authentication. The pairing code is scoped to the user's authenticated session.

### Pairing Code Details
- Format: `HOSTNAME-XXXX` (e.g., `MacBook-Pro-7X2`)
- Generated by dashboard when user clicks "Add laptop"
- Valid for 10 minutes
- One-time use — invalidated after first successful pair
- Displayed in dashboard AND copy-able via CLI flag `--copy-code`

### Pairing Details

- Device ID: generated by bridge on first run, stored locally
- Device code: 6-char alphanumeric shown in dashboard, typed into bridge CLI (fallback)
- OR: bridge auto-activates via localhost redirect (primary path)
- Refresh token: stored at `~/.conductor/bridge-refresh-token` (0600)
- Device name: laptop hostname, user can rename in dashboard

## Pairing Without Browser on Laptop (Alternative)

For headless servers or users who prefer CLI:

```
$ conductor-bridge pair --code ABC123
Connecting to relay.conductor.dev...
Device paired: "headless-server"
Connected.
```

## Multi-Device Support

- One dashboard account → many paired laptops
- User names each laptop in the dashboard
- Sessions can target a specific laptop or auto-select the only online laptop
- "Connect to..." dropdown in session view

## Terminal Behavior

- ttyd runs on laptop (already implemented)
- Bridge proxies ttyd WebSocket through relay to dashboard
- No middle WebSocket — bridge terminates at laptop's local ttyd
- Terminal feels like a direct connection

## Security

- Bridge connects OUTBOUND only — no inbound ports opened on laptop
- TLS 1.3 minimum for all connections
- Refresh token stored with filesystem permissions (0600)
- Device can be revoked from dashboard (revokes refresh token)
- Bridge auto-disconnects if token is revoked

## What to Build

### Phase 1 — Minimal Viable Bridge

1. **Bridge binary** in Go (single binary, no deps)
   - `conductor-bridge daemon` — background mode
   - `conductor-bridge pair --code CODE` — headless pairing
   - Auto-reconnect with backoff
   - Store refresh token to `~/.conductor/bridge-refresh-token`

2. **Device authorization server** (extend existing relay)
   - `GET /device/code` — returns device code + user code
   - `POST /device/token` — exchange device code for tokens
   - Store device → refresh token mapping

3. **Dashboard pairing page**
   - Show device code (e.g., "MacBook-Pro-42X")
   - "Connect this laptop" button
   - Confirm screen after pairing

4. **Bridge tunnel to relay**
   - WebSocket from bridge to relay
   - Authenticated via device refresh token
   - Proxy terminal sessions through relay

### Phase 2 — Full Dashboard Integration

5. **Device list in dashboard**
   - Show all paired devices with online/offline status
   - Rename device, revoke device
   - "Add new device" shows pairing code/instructions

6. **Terminal via bridge**
   - Replace direct ttyd with bridge-proxied ttyd
   - User selects which laptop to use for a session

## Out of Scope (v1)

- File browser in dashboard
- Session sharing / read-only links
- Multiple users sharing one laptop
- Mobile native app (web is universal access)

## Stack

- **Bridge:** Go (single binary, no runtime deps, cross-compiles to all platforms)
- **Relay:** Rust (already have the foundation in `conductor-relay`)
- **Dashboard:** Next.js + TypeScript (already have this)
- **Auth:** GitHub OAuth for dashboard, device tokens for bridge
